Порядок процесса "Init Session Webhook" (/api/v1/tcs/initsession):
1. Обработка запроса на начало проверки
	а. Создание папки сессии (если есть, то удаление старой)
	б. Скачивание макета 
	в. Запрос информации о продукте
	г. Формирование json содержаший полученные данные по запросу (требуется кодирование)
	д. Сохранение данных в папке сессии: макет, json параметров
	е. Выставляем сигнал начала процесса: 3.~
	ж. Передача в горячую папку файла json для начала обработки

	- Возврат кодов, обработка ошибок, результат выполнения:
		а. Возврат запросу результата (не требует кодирования): 1 - успешно, иначе ошибка. На этом этате внятного сообщения клиенту нет. В случае ошибки выводим сообщение со стандартной заглушкой (логика на стороне TCS)
		б. Проверка что процесс уже запущен (наличие файлов и сигналов): Ничего не делаем, но возврашаем успешный ответ на запрос (1)

2. Начало обработки на основании данных параметров (разделение процесса на два параллельных) json из п.1 "TCS Preflight":
	а. Открытие файла параметров, получение необходимой информации о макете, создание каталогов для превью
	б. Выставляем сигнал начала генерации превью: 1.~
	в. Выставляем сигнал начала процесса получения информации о макете: 2.~
	г. Удаляем сигнал начала процесса обработки: 3.~
	е. Отправка по коннектору макета для генерации превью
	ж. Отправка по коннектору макета для пролучения информации
	* Далее процессы веток "е" и "ж" протекают асинхронно
	
	- Обработка ошибок, результат выполнения:
		а. Любая ошибка в процессах "а"-"ж" должна остановить выполнение проверки уведомить об этом систему TCS
		б. Единственная ошибка, предполагающая внятное уведомление для клиента - это некорректный или поврежденный файл макета, иначе выводим сообщение со стандартной заглушкой
		в. Запись сообщение ошибки в отдельный сигнальный файл: 4.~. При этом, удалим любые другие сигналы

3. Анализ PDF файла макета, получаеного из п.2.ж "Create Report To TCS":
	а. Создание файла конфигурации Pitstop
	б. Выполнение процесса Pitstop
	в. Ожидание завершения генерации превью макета
	г. Генерация превью уменьшенных размеров
	д. Удаление сигнала процесса получения информации: 2.~
	е. Выставляем сигнал успешного завершения всех процессов: 0.~

	- Обработка ошибок, результат выполнения:
		а. Любая ошибка в процессах "а"-"е" должна остановить выполнение проверки уведомить об этом систему TCS
		б. Запись сообщение ошибки в отдельный сигнальный файл: 4.~. Удаление сигнала процесса получения информации происходит в любом случае: 2.~

4. Обработка запроса на получение результата проверки "WebHooks TCS Preflight":
	а. Просмотр на предмет наличия сигналов в следующем порядке:
		а.1. Сигнал на принудительное завершение (4.~). Прервать обработку и передать сообщение из сигнала в TCS
		а.2. Процессы успешно завершены (0.~). Обработка завершена, передаем отчет
			- Открытие файла параметров
			- Открытие файла отчета
			- Анализ отчета и формирование ответа по ошибкам
			- Добавление в ответ превью файлов по всем страницам
			- Статус прогресса 100%
		а.3. Начало работы (3.~). Физически процессы еще не начались. На этот момент: скачен макет, горячая папка забрала файл параметров, необходимые параметры проверены и все готово для начала процессов
			- Статус прогресса 10%. Основные затраты времени на данном этапе - загрузка макета и передача сигнала в скрипт
		а.4. Процесс проверки физически начался, но еще не завершен (2.~). Параллельно происходит процесс генерации превью, на который пока не обращаем внимание
			- Статус прогресса 15%. Прибавляем к предыдущему состоянию (а.3.) 5% на то, что произошел процесс передачи работы между скриптами
		а.5. Процесс проверки физически завершен, но генерация превью продолжается (1.~)
			- Статус прогресса (50 + х)%. 50% на завершения проверки (а.4.) + %, котрый учитывает сколько превью успело нагенериться в пределах 50-95%
		а.6. Не нашлось ни одного сигнала а директории сессии.
	- Обработка ошибок, результат выполнения:
		а. Любая ошибка в процессах должна остановить выполнение проверки уведомить об этом систему TCS
		б. Единственная ошибка, предполагающая внятное уведомление для клиента - это сообщение из сигнала 4.~
	- Возврат кодов
		а. 1  - Процесс успешно завершен. Возврат результатов проверки
		б. 0  - Процесс еще не завершен. Возврат статуса текущего состояния в %%
		в. -1 - Обработанная ошибка. Возврат сообщения для вывода клиенту
